# 10 — ADR Draft Set v0.9

These ADRs formalize the product and architecture decisions made so far. They preserve the core method: define one business unit, measure full workflow cost, verify the outcome, and compare cost against value or manual baseline.

## ADR-001 — Use `us-east-1` as the deployment region

Status: Accepted.

Decision: Deploy the MVP in `us-east-1`.

Rationale: The user selected `us-east-1`, and AgentCore supports that region.

Consequences: All CDK stacks, S3 buckets, DynamoDB tables, Lambda functions, AgentCore resources, Bedrock model calls, and CloudWatch resources default to `us-east-1`.

## ADR-002 — Use AWS CDK in TypeScript for infrastructure

Status: Accepted.

Decision: Use AWS CDK TypeScript for infrastructure.

Rationale: The product stack is TypeScript-first.

Consequences: Infrastructure lives under `/infra` with stack outputs used to configure frontend, API, runtime, and tool Lambdas.

## ADR-003 — Prefer CloudFormation L1 AgentCore constructs

Status: Accepted.

Decision: Use CloudFormation/L1 AgentCore resources unless a stable higher-level CDK construct materially reduces complexity.

Rationale: AgentCore is evolving. L1 resources are less likely to hide unstable assumptions.

Consequences: AgentCore resources must be isolated in `AgentCoreStack`. If alpha constructs are used, pin package versions.

## ADR-004 — Use AgentCore Runtime as the agent execution environment

Status: Accepted.

Decision: Run the Strands stage coordinator in Amazon Bedrock AgentCore Runtime.

Rationale: The product is explicitly about measuring an agentic workflow on AWS AgentCore.

Consequences: The Control API creates a `Run`, then invokes AgentCore Runtime with `workspaceId`, `documentId`, `jobId`, and `runId`.

## ADR-005 — Use TypeScript Strands as the primary agent implementation

Status: Accepted.

Decision: Implement the AgentCore Runtime app in TypeScript using Strands.

Rationale: The project is TypeScript-first and TypeScript Strands is the selected direction.

Consequences: The agent should be mostly plain TypeScript stage-runner code, with Strands as the AgentCore-compatible layer.

## ADR-006 — Keep a Python fallback for PDF tooling only

Status: Accepted.

Decision: Use TypeScript wherever practical, but allow Python for PDF extraction, image extraction, and PDF recomposition if TypeScript libraries are materially weaker.

Rationale: PDF coordinate extraction and recomposition are library-sensitive.

Consequences: Tool implementation language is hidden behind AgentCore Gateway.

## ADR-007 — The Control API creates Runs; the agent executes them

Status: Accepted.

Decision: The Control API owns `Run` creation. AgentCore Runtime executes an existing `Run`.

Rationale: State transitions, attempt numbering, and user-facing consistency are easier to control when the API creates the durable `Run` record.

Consequences: The agent receives IDs, loads context from repositories, executes stages, and updates the existing run.

## ADR-008 — Use AgentCore Gateway as the tool boundary

Status: Accepted.

Decision: Expose deterministic document-processing operations through AgentCore Gateway.

Rationale: Gateway provides the tool-connectivity layer and supports Lambda-backed targets.

Consequences: Tools are grouped into `PdfPipelineTools`, `TranslationTools`, and `EvaluationTools` Gateway targets.

## ADR-009 — Use Lambda-backed Gateway targets for MVP tools

Status: Accepted.

Decision: Implement Gateway tools as Lambda targets.

Rationale: Required tools are custom deterministic operations.

Consequences: Tool schemas live in the repo; Gateway-exposed tool names must account for target-name prefixes.

## ADR-010 — Use deterministic stage plans, not unconstrained agent behavior

Status: Accepted.

Decision: The agent executes fixed workflow stage plans for V1, V2, and V3.

Rationale: Cost attribution, repeatability, and stage-level auditability require deterministic stage plans.

Consequences: The agent should not accept arbitrary natural-language instructions during a run.

## ADR-011 — Pass S3 artifact references, not raw PDF bytes

Status: Accepted.

Decision: Store PDFs and generated outputs in S3. Pass artifact IDs and S3 keys through requests.

Rationale: The workflow creates multiple durable outputs.

Consequences: AgentCore Runtime, Gateway tools, and APIs receive artifact references, not raw document bytes.

## ADR-012 — Use a private S3 artifact bucket per environment

Status: Accepted.

Decision: Use one private, encrypted S3 artifact bucket per environment.

Rationale: Source PDFs, translated PDFs, image assets, and evaluation reports should not be public.

Consequences: Artifact access goes through presigned URLs generated by the Control API.

## ADR-013 — Use separate DynamoDB tables for MVP clarity

Status: Accepted.

Decision: Use separate DynamoDB tables for each major entity instead of a single-table design.

Rationale: Access patterns are simple, and primary implementation risk is workflow/cost instrumentation.

Consequences: Tables include `Documents`, `TranslationJobs`, `Runs`, `StageEvents`, `Artifacts`, `LedgerItems`, `EvaluationResults`, `ReviewDecisions`, `PriceBooks`, and `AppSettings`.

## ADR-014 — Model `TranslationJob` as the business unit

Status: Accepted.

Decision: A `TranslationJob` represents the business task: produce one accepted English PDF from one Spanish source PDF.

Rationale: Economics require measuring cost per accepted business task, not cost per technical run.

Consequences: Job-level economics aggregate all attempts, review decisions, retries, remediations, and accepted outcomes.

## ADR-015 — Model `Run` as a technical attempt

Status: Accepted.

Decision: A `Run` is one technical execution attempt for a `TranslationJob`.

Rationale: Failed, rejected, retried, and remediated attempts must count toward the cost of the business task.

Consequences: Multi-attempt jobs include all run costs when calculating cost per verified outcome.

## ADR-016 — Persist workflow progress as `StageEvents`

Status: Accepted.

Decision: Every executed stage creates a `StageEvent`.

Rationale: Stage-level records make the workflow auditable and allow costs, artifacts, warnings, tool calls, and failures to be attributed.

Consequences: UI timeline reads from persisted `StageEvents`, not transient logs.

## ADR-017 — Persist durable outputs as `Artifacts`

Status: Accepted.

Decision: Every durable output is represented by an `Artifact` record.

Rationale: Source PDFs, inspection JSON, layout JSON, chunks, translated chunks, image assets, final PDFs, previews, and evaluation JSON must be addressable.

Consequences: S3 stores object bytes; DynamoDB stores artifact metadata.

## ADR-018 — Use `LedgerItems` as the source of truth for economics

Status: Accepted.

Decision: Product economics are calculated from `LedgerItems`.

Rationale: Logs/traces are for debugging and reconciliation; economics need normalized cost records.

Consequences: The ledger table is authoritative for run cost, job cost, cost per verified outcome, and unit margin.

## ADR-019 — Use a `PriceBook` for cost calculation

Status: Accepted.

Decision: Store pricing assumptions in a `PriceBook` table.

Rationale: Model/tool/runtime prices can change. Cost assumptions should be visible and editable.

Consequences: Ledger rows must include `priceBookVersion`.

## ADR-020 — Separate LLM-only cost from full workflow cost

Status: Accepted.

Decision: UI must show `LLM-only cost` and `full workflow cost` separately.

Rationale: Token cost is only one row in the product-cost ledger.

Consequences: `llmOnlyCostUsd` comes only from `MODEL_INFERENCE` ledger rows. `fullWorkflowCostUsd` sums all ledger rows.

## ADR-021 — Human review is a costed economic event

Status: Accepted.

Decision: Reviewer decisions create `ReviewDecision` records and `HUMAN_REVIEW` ledger rows.

Rationale: Human review is part of full workflow cost.

Consequences: Accept, reject, and escalate actions require reviewer time and update job economics.

## ADR-022 — Acceptance is distinct from automated evaluation

Status: Accepted.

Decision: Automated evaluation does not by itself create a verified business outcome.

Rationale: Verified outcome is a reviewer-accepted translated PDF.

Consequences: A run can complete evaluation and remain `AWAITING_REVIEW`.

## ADR-023 — Rejected work still counts as cost

Status: Accepted.

Decision: Rejected runs retain full workflow cost but do not produce a verified outcome.

Rationale: Cost per verified outcome must include costs of failed/rejected work when evaluating business viability.

Consequences: Rejected jobs show full cost but `costPerVerifiedOutcomeUsd = null`.

## ADR-024 — Use Amazon Bedrock Converse through a shared wrapper

Status: Accepted.

Decision: All Bedrock model calls use a shared Converse wrapper.

Rationale: Converse provides a consistent interface, request metadata, and usage information for token accounting.

Consequences: Translation and evaluation tools must not call Bedrock directly outside the wrapper.

## ADR-025 — Enforce structured JSON model outputs

Status: Accepted.

Decision: Translation, image-translation, and evaluation prompts require valid JSON matching Zod schemas.

Rationale: Workflow needs deterministic parsing, chunk alignment, and reliable records.

Consequences: Malformed model output triggers bounded repair and creates retry/cost evidence.

## ADR-026 — Use a static glossary for MVP

Status: Accepted.

Decision: Use a static Spanish-to-English glossary for the controlled document.

Rationale: MVP needs terminology consistency without glossary-management subsystem.

Consequences: Required terms include `reembolso → refund`, `elegibilidad → eligibility`, `contracargo → chargeback`, `revisión manual → manual review`, and `caso escalado → escalated case`.

## ADR-027 — Use controlled digitally generated PDFs for MVP

Status: Accepted.

Decision: Limit MVP support to controlled, digitally generated Spanish PDFs.

Rationale: Arbitrary scanned PDFs require OCR and robust layout handling outside the core economics method.

Consequences: Scanned PDFs are marked unsupported or warned as outside MVP scope.

## ADR-028 — Use approximate PDF recomposition

Status: Accepted.

Decision: Recompose translated PDFs with readable approximate layout preservation.

Rationale: Product needs a usable translated artifact, not desktop-publishing fidelity.

Consequences: Layout overflow warnings are acceptable and surfaced in evaluation.

## ADR-029 — Implement V1 as text-only baseline

Status: Accepted.

Decision: V1 translates extractable text only.

Rationale: V1 establishes lowest-complexity baseline for cost, quality, and acceptance.

Consequences: Embedded image text may remain Spanish and appear as evaluation warning.

## ADR-030 — Implement V2 as text plus image-text annotation

Status: Accepted.

Decision: V2 translates extractable text and image text using annotations, callouts, captions, or overlays.

Rationale: V2 demonstrates added capability and cost impact without image inpainting.

Consequences: V2 creates additional tool/model ledger rows compared with V1.

## ADR-031 — Implement V3 as architecture optimization

Status: Accepted.

Decision: V3 optimizes workflow through routing, selective image handling, batching, and skipped non-material work.

Rationale: Architecture determines margins.

Consequences: V3 should show avoided work compared with V2, based on real ledger rows.

## ADR-032 — Use deterministic plus model-based evaluation

Status: Accepted.

Decision: Evaluation combines deterministic checks with model-based semantic scoring.

Rationale: Deterministic checks catch structural failures; model evaluation scores semantic coverage and quality.

Consequences: Deterministic failures can override model optimism.

## ADR-033 — Use AgentCore Observability and CloudWatch, but do not derive economics only from logs

Status: Accepted.

Decision: Use AgentCore Observability and CloudWatch for telemetry and reconciliation while keeping `LedgerItems` as product source of truth.

Rationale: Operational telemetry is not enough for normalized product economics.

Consequences: `Run`, `StageEvent`, and `LedgerItem` records carry trace/span IDs where available.

## ADR-034 — Label cost basis honestly

Status: Accepted.

Decision: Every cost display labels whether it is telemetry-derived, price-book-estimated, bill-reconciled, or mixed.

Rationale: Do not imply exact AWS bill attribution unless reconciliation exists.

Consequences: Default MVP label is `TELEMETRY_DERIVED_PRICE_BOOK_ESTIMATE`.

## ADR-035 — Defer AWS bill reconciliation

Status: Accepted.

Decision: Do not implement real-time AWS bill reconciliation in MVP.

Rationale: Core product is full workflow cost instrumentation and cost-per-outcome calculation.

Consequences: Future reconciliation worker may update ledger rows to `AWS_BILL_RECONCILED`.

## ADR-036 — Do not provision AgentCore Memory in MVP

Status: Accepted.

Decision: Keep AgentCore Memory disabled unless glossary/reviewer-preference memory is added.

Rationale: The document itself is not memory; MVP can use static glossary.

Consequences: Memory pricing and ledger rows remain structurally supported but unused.

## ADR-037 — Include AgentCore Policy as narrow governance feature, not critical-path blocker

Status: Accepted.

Decision: Implement AgentCore Policy only for narrow tool-access rules once main workflow is stable.

Rationale: Policy supports AgentCore framing but should not block first end-to-end workflow.

Consequences: First policy should restrict `translate_image_text` to V2/V3 or enforce image-count thresholds.

## ADR-038 — No product replay, synthetic-run, or recording mode

Status: Accepted.

Decision: Product performs real runs and stores normal historical results. Recording is external.

Rationale: Product exists to demonstrate and measure a real workflow, not to be a presentation system.

Consequences: No `LIVE_CAPTURE`, `REPLAY_CAPTURED`, `SYNTHETIC_SEED`, or presentation mode.

## ADR-039 — Use historical comparison groups for V1/V2/V3

Status: Accepted.

Decision: Compare architecture variants using real historical jobs linked by `comparisonGroupId`.

Rationale: V1, V2, and V3 are alternative workflows, not replay modes.

Consequences: Comparison reads persisted jobs, runs, evaluations, review decisions, and ledger rows.

## ADR-040 — Use development fixtures only as scaffolding

Status: Accepted.

Decision: In-memory repositories and fixtures are allowed for development/tests, not product modes.

Rationale: Fixtures help build UI/costing before AWS integration without fake product data.

Consequences: Fixture code must be isolated from production paths.

## ADR-041 — Build in vertical slices

Status: Accepted.

Decision: Build economic correctness first, then persistence, then AgentCore, then real PDF/Bedrock processing.

Rationale: Product value depends on ledger and outcome model.

Consequences: Codex should implement schemas, costing, repositories, UI, and API shape before full AgentCore/PDF complexity.

## ADR-042 — Keep frontend recording-neutral

Status: Accepted.

Decision: UI should be clear and useful as a product, without recording/presentation mode.

Rationale: Product should not be shaped around recorded-video format.

Consequences: Good visual hierarchy and accessibility are requirements; presentation-specific behavior is not.

## ADR-043 — Use Next.js / React / TypeScript for frontend

Status: Accepted.

Decision: Build frontend with Next.js or React in TypeScript.

Rationale: Aligns with TypeScript-first stack.

Consequences: Frontend routes include Documents, Jobs, Runs, Result, Evaluation, Ledger, Comparison, and Economics Settings.

## ADR-044 — Use API Gateway plus Lambda for Control API

Status: Accepted.

Decision: Use API Gateway HTTP API with TypeScript Lambda Control API.

Rationale: Backend API is CRUD, presigned URLs, state transitions, and AgentCore invocation.

Consequences: Control API should not do long-running document processing.

## ADR-045 — Keep long-running processing in AgentCore/tools, not Control API

Status: Accepted.

Decision: Long workflow execution belongs in AgentCore Runtime and Gateway tools.

Rationale: Control API should remain responsive.

Consequences: Starting run creates records and invokes agent; polling reads persisted state.

## ADR-046 — Do not use VPC networking for MVP

Status: Accepted.

Decision: Do not place Lambda or AgentCore Runtime in a VPC unless private dependency requires it.

Rationale: MVP uses public AWS endpoints with IAM and no private database.

Consequences: Avoid VPC cold-start/networking complexity.

## ADR-047 — Use least-privilege IAM roles

Status: Accepted.

Decision: Create separate IAM roles for Control API, Agent Runtime, PDF tools, Translation tools, Evaluation tools, and optional reconciliation.

Rationale: Each component has different access requirements.

Consequences: Avoid broad wildcard permissions except temporary implementation spikes.

## ADR-048 — Use ECR container images where native dependencies require them

Status: Accepted.

Decision: Use container images for AgentCore Runtime and for PDF tools if native PDF libraries require them.

Rationale: Native PDF dependencies may be difficult to package as Lambda zip.

Consequences: Image tags should be pinned by build ID or commit SHA.

## ADR-049 — Seed only configuration, not fake product runs

Status: Accepted.

Decision: Seed active price book and optionally controlled documents through normal ingestion; do not seed fake product-facing run histories.

Rationale: Do not blur real workflow evidence with manufactured run data.

Consequences: Demo validation uses real runs against controlled document.

## ADR-050 — Use a controlled Spanish refund-eligibility PDF as primary test fixture

Status: Accepted.

Decision: Use `Procedimiento de Reembolsos y Elegibilidad`.

Rationale: Supports business-value framing, glossary consistency, table/layout extraction, and image-text handling.

Consequences: V1/V2/V3 can be compared on stable input.

## ADR-051 — Defer scanned-PDF OCR support

Status: Accepted.

Decision: Do not implement full OCR for arbitrary scanned PDFs in MVP.

Rationale: OCR is valuable but orthogonal to unit-economics method.

Consequences: Inspection detects likely scanned pages and marks unsupported or warns.

## ADR-052 — Defer production-grade authentication

Status: Accepted.

Decision: Do not build enterprise auth/RBAC for MVP.

Rationale: Auth is not central to economic method.

Consequences: Dev deployment should still be protected appropriately, but complex auth is future scope.

## ADR-053 — Keep charts simple

Status: Accepted.

Decision: Use simple tables and bar charts for comparison and cost breakdown.

Rationale: Key product value is accurate attribution, not analytics sophistication.

Consequences: Prioritize clarity over visual complexity.

## ADR-054 — Use Cost per Verified Outcome as a metric, not the full product message

Status: Accepted.

Decision: Frame unit economics as value versus full cost, with cost per verified outcome as cost-side metric.

Rationale: First question is whether the problem is worth solving, not which model is cheapest.

Consequences: UI should show value, full cost, cost per verified outcome, and unit margin together.

## ADR-055 — Treat AgentCore Policy, Memory, and runtime reconciliation as extensibility points

Status: Accepted.

Decision: Keep schemas and ledger component types for Policy, Memory, runtime reconciliation, but do not force them into first working slice.

Rationale: Full AgentCore economics story matters, but first end-to-end path should stay focused.

Consequences: MVP can land V1 first, then add V2, V3, Policy, and reconciliation incrementally.

## ADR-056 — A run that fails automated evaluation is still reviewable

Status: Accepted.

Decision: Technically completed run with failed evaluation goes to `AWAITING_REVIEW`, not automatically `FAILED`.

Rationale: `FAILED` means technical workflow failure. Evaluation failure is a business/review decision.

Consequences: Reviewers can reject or escalate evaluation-failed runs; cost remains visible.

## ADR-057 — Use trace IDs for correlation, not primary identity

Status: Accepted.

Decision: Use `runId`, `jobId`, `documentId` as primary product identifiers; trace/span IDs for operational correlation.

Rationale: Trace IDs can vary; product records need stable identifiers.

Consequences: Logs, StageEvents, LedgerItems include product IDs and trace/span IDs where available.

## ADR-058 — Keep model IDs configurable

Status: Accepted.

Decision: Translation and evaluation model IDs are configuration values, not hard-coded.

Rationale: Model availability, pricing, and suitability can change.

Consequences: Model IDs are recorded in ledger/evaluation rows.

## ADR-059 — Use generated page previews as reliability fallback

Status: Accepted.

Decision: Browser-native PDF rendering is acceptable, but pipeline should generate page preview PNGs.

Rationale: Browser PDF rendering can be inconsistent.

Consequences: `PDF_PREVIEW_PNG` artifacts are part of artifact model.

## ADR-060 — Keep remediation visible if implemented

Status: Accepted.

Decision: Remediation/follow-up attempts must be represented as additional cost.

Rationale: Remediation directly affects cost per verified outcome.

Consequences: Remediation may create new runs, retry ledger rows, or `REMEDIATION` ledger rows.
